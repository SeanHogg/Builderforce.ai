## Platform Roadmap (umbrella — Builderforce.ai · BuilderForce Agents · Evermind/SSM)

> **Vision**: Human as the manager — developers supervise, prioritize, approve, and steer AI agents rather than writing code themselves.

### 🌟 North Star — "Define a need, the agentic system solves it"

> Any human defines a need in any modality (prose · dataset/docs · process chart · persona · diagnostic findings) and the platform compiles it into an agent that runs on the right surface (IDE · desktop · cloud · workflow step). The unifying primitive: `compile(need, modality) → AgentSpec → deploy(AgentSpec, surface)`. The spine (`AgentSpec` IR + `lowerAgentSpec` + `compile()`/`deploy()` registries + `PolicyGate` + all six modality adapters + `/compile`) is **shipped**; remaining work is enforcement depth + data providers (below). Full design: **[PRD-agent-compile-primitive.md](./PRD-agent-compile-primitive.md)**.

## 15 · 🛠️ Platform — DB, CI/CD, Migrations, Cost & Tech-debt

### 💤 Neon compute cost — KV cron work-gate residuals

- **Idle-platform scheduled tasks coarsen to the 30-min floor.** When NO ticket activity has signalled work, time-scheduled sweeps (`runDueTriggers`, `runDueReports`, `runDueCeremonies`, `dueSnapshots`, board-sync polls) only fire on the `FLOOR_INTERVAL_MS` (30 min) safety sweep, so a workflow scheduled finer than 30 min can be up to ~30 min late while idle. Fix: **dynamic next-due gate** — after each active tick, `SELECT MIN(next_run_at)` across the schedule tables and publish `cron:next-due` to KV; the gate also runs when `now >= nextDue`. Unblocks: exact scheduled-workflow precision with maximal autosuspend.
- **Floor interval is a hardcoded constant, not env-tunable.** `FLOOR_INTERVAL_MS = 30min` sets the idle wake cadence (~48 wakes/day ≈ ~120 compute-hrs/mo, under the Neon Free ~191-hr ceiling). Fix: read an optional `CRON_FLOOR_INTERVAL_MS` env override.
- **Phase 0 is a manual console action.** Autosuspend/scale-to-zero + 0.25 CU min must be enabled in the Neon console for the gate to actually save money. Not code; tracked here so it isn't lost.
- **Storage was OVER the Free 512 MB ceiling (measured 832 MB) — root cause is BLOAT.** `manager_actions` is 593 MB but holds only 46k rows (~24 MB of real data, 0 dead tuples): ~500 MB is reclaimable page bloat autovacuum never returned to the OS. Retention (`manager_actions` 30d, `tool_audit_events` 90d) caps future growth but reclaims nothing here; the immediate reclaim needs a one-time **`VACUUM (FULL, ANALYZE) manager_actions;`**. BLOCKED by the Claude Code permission classifier (mutating prod-DB write in auto mode) — pending the operator running it or granting a Bash allow rule. CAVEAT: on Neon the logical size drops immediately but BILLED storage lags until the PITR window rolls past the rewrite. Unblocks: ~832 MB → ~320 MB, back under Free.
- **`manager_actions` re-bloats because the every-5-min manager sweep inserts feed rows across all tenants.** Consider (a) skipping the `detail` payload for feed-only (`run_task_id IS NULL`) cron decisions, and (b) a periodic autovacuum/`VACUUM` schedule.
- **`autovacuum_vacuum_scale_factor` default (0.2) is too lax for a high-write feed.** A per-table `ALTER TABLE manager_actions SET (autovacuum_vacuum_scale_factor=0.02, fillfactor=90);` would keep it tight. Control-plane/DDL — pending operator.
