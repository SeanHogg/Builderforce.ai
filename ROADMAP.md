## Platform Roadmap (umbrella — Builderforce.ai · BuilderForce Agents · Evermind/SSM)

> **Vision**: Human as the manager — developers supervise, prioritize, approve, and steer AI agents rather than writing code themselves.

### 🌟 North Star — "Define a need, the agentic system solves it"

> Any human defines a need in any modality (prose · dataset/docs · process chart · persona · diagnostic findings) and the platform compiles it into an agent that runs on the right surface (IDE · desktop · cloud · workflow step). The unifying primitive: `compile(need, modality) → AgentSpec → deploy(AgentSpec, surface)`. The spine (`AgentSpec` IR + `lowerAgentSpec` + `compile()`/`deploy()` registries + `PolicyGate` + all six modality adapters + `/compile`) is **shipped**; remaining work is enforcement depth + data providers (below). Full design: **[PRD-agent-compile-primitive.md](./PRD-agent-compile-primitive.md)**.

### 🚨 Incident Management — residual polish (post 2026-07-11 full-slice ship)

> The incident-management subsystem shipped 2026-07-11 (Help Desk / Incident Manager agent, Freshdesk connector, incident bridge, on-call rotations, timed escalation, Teams/Slack/email paging, war-room; see DONE.md). These are the non-blocking follow-ups; each has a live/working path today.

- **War-room is a persisted feed, not yet a live room.** `IncidentService.ensureWarRoom` creates a `brain_chats` row (`origin='incident'`) linked as `prod_incidents.war_room_chat_id`, and the incident timeline (`incident_events`) is the war-room feed. It is NOT yet wired to real-time presence via `CeremonyRoomDO` (`incident:<id>` room) nor surfaced as a joinable Brain chat in the incidents UI. Wiring the chat view + a `CeremonyRoomDO` room would make it a true live war room with typing/presence. Unblocks synchronous humans+agents collaboration during an incident.
- **Incident-triage runs don't guard board-task lane movement on completion.** Unlike the Validator review marker (which skips the status transition to avoid a re-trigger loop), an `incidentTriage` run completing on the bridged INCIDENT board task lets the normal COMPLETED→next-lane path move it. Harmless today (no on-status incident trigger to loop), but the board lane can drift from `prod_incidents.status`. Fixing = have the runtime honor `isIncidentTriagePayload` to skip the lane transition, and mirror `prod_incidents.status`→lane on `IncidentService.updateIncident`.
- **On-call notification for agent members is timeline-only.** A `c:<agentRef>` on-call member has no external channel (Teams/Slack/email); `incidentNotifier` logs a timeline note but does not dispatch the agent. Wiring `contact:`/`u:` covers humans; agents-on-call could instead trigger `dispatchIncidentTriage`. Unblocks an agent being a first-class on-call responder.
- **Freshdesk push-back is title/description only.** `FreshdeskBoardProvider.pushUpdate` mirrors Freshservice (subject/description); it does not sync incident status/severity back to the Freshdesk ticket. Unblocks bidirectional help-desk sync (resolve in BF → close in Freshdesk).
- **Tenant-wide incidents (no projectId) don't feed Evermind.** `recordIncidentLearning` is project-scoped (Evermind has no tenant-only corpus), so an incident opened without a project contributes no lesson. Unblocking needs a tenant-level learning corpus or attaching such incidents to a default project. The RCA is still published to Knowledge regardless.
- **No general `knowledge.create` MCP tool.** Agents can author Knowledge only via `incidents.postmortem`; there's no tool for an agent to write a standalone SOP / runbook / known-error article. Adding `knowledge.create` (author + publish via the shared `publishKnowledgeDoc`) would let PM/Service-Desk agents build the KB directly. Unblocks agent-authored runbooks that `knowledge.search` then recalls during triage.

### 🔐 Login "CORS error for everyone" — investigate the Cloudflare-level trigger

> Reported 2026-07-09: users could not log in (any method); browser console showed `net::ERR_FAILED` + "No 'Access-Control-Allow-Origin' header is present" on **every** endpoint (`/health`, `/api/auth/legal/current`, `/api/auth/web/login`). From outside the worker is healthy — curl (incl. fully browser-shaped requests) gets `Access-Control-Allow-Origin: *` on every method + preflight, so this is NOT a worker CORS-config bug. A headerless response reaching the browser means the response came from a **Cloudflare-level failure** (Worker exception 1101, CPU/mem 1102, or a WAF/Bot-Fight/rate-limit challenge page), which is served without the worker's CORS headers.
> - **Mitigated (api 2026.7.67):** the top-level `fetch` handler (`api/src/index.ts`) now wraps `buildApp(env).fetch(...)` in try/catch and returns a **CORS'd JSON 500** — so a construction/dispatch throw is now a readable error in the browser instead of a phantom "CORS error for everyone". This converts the invisible failure into an observable one; it does not remove the trigger.
> - **Still open — find the actual trigger:** (a) check Cloudflare dashboard → Workers logs / Logpush for `1101/1102` exceptions and the `[fetch:top-level]` / `[unhandled]` console lines around the incident window; (b) check `api_error_log` table + the Product Quality pillar for the real 500 message; (c) confirm no **Bot Fight Mode / WAF / rate-limit** rule is challenging browser (cookie-bearing, residential-IP) traffic while letting datacenter curl through; (d) rule out a transient **Neon** connection-construction failure in `buildDatabase`/`buildApp` under cold-start/connection pressure. Unblocks: root-cause + permanent fix for the login outage class.

### 🛡️ System Audits (SOC 2 / Architecture / Quality / PM Vision / Privacy)

- **P2 — Agent audit deep-pass does not re-score/enrich the report.** `AuditRunner.runAudit` (`api/src/application/tools/AuditRunner.ts`) always produces the authoritative report from the deterministic `scan()` and files a best-effort remediation ticket for the audit agent (security_audit / quality_audit / pm_vision_audit / privacy_audit workflows). Unlike the Architecture diagnostic (which `AnalysisRunnerDO` re-scores from a `principles` artifact on completion), there is **no completion hook** that parses the SOC 2 / Quality / PM / Privacy agent's structured findings back into an updated `ToolResult`. So the report reflects the deterministic scan, not the agent's deep content-level findings, until the remediation PR merges and the audit is re-run. Fix = have the audit workflow emit a structured findings artifact + an `AuditRunnerDO`-style completion hook that calls `recordExternalRun` with the enriched result. Unblocks: agent-depth scoring, not just heuristic scoring.
- **P3 — Live agent audit run unverified in-repo (needs creds + runtime).** The board ticket → `maybeAutoRunOnLaneEntry` → `dispatchCloudRunForTask` wiring is in place and typechecks, but a full end-to-end live audit run needs a connected repo credential + an available cloud runtime, which are not exercisable in the dev environment. The deterministic path (and its unit tests) fully cover scoring. Fix = verify the live dispatch against a real connected repo. Unblocks: confirmed agent remediation PRs from onboarding.
- **P3 — Privacy audit signals are path-based only.** `privacyScan` + `AuditRunner` privacy heuristics (GDPR/CCPA/CAN-SPAM) detect a privacy policy / cookie consent / unsubscribe / data-export / erasure / retention surface from **file paths**, not content — so a route that exists but doesn't actually gate trackers or truly delete data still scores as present. The `privacy_audit` agent workflow covers the content-level review, but the deterministic first-pass can over-credit. Fix = read key files (email templates, consent init, deletion handlers) for content signals. Unblocks: fewer false-positives in the instant report.
- **P3 — Per-gap audit re-runs duplicate tickets (no dedup).** `AuditRunner.runAudit` now files one remediation ticket per gap for `ticketPerFinding` audits (the Privacy diagnostic), each independently resolvable — but re-running the audit before a gap is fixed re-creates a duplicate ticket for that gap (same behaviour as the Security agent's `record_finding`). Unblocks: clean re-runs during dogfooding without board spam.

Fix = persist the filed `taskId`s on the run (or join project tasks by `${audit.name}` title) and derive a real "remediation PR opened/merged" badge from the task's PR/branch state, then render it in the strip's gauge card. Unblocks: the true remediation-status badge on the card + analytics, matching the marketing visual.
- **P3 — Diagnostic strip is manager-gated (hidden for non-managers).** The card/table diagnostics strip reads `GET /api/tools/rollup` and the analytics gauges read `GET /api/tools/projects/:id/score`, both `requireRole(MANAGER)`; a non-manager gets 403 and the strips self-hide (same gating as the existing maturity block).
