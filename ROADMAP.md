### ­¤¦Ā Brain chat tool/memory steps are not DURABLE ŌĆö they vanish on reload

> Observed 2026-07-09 (operator): a memory write / tool call the Brain agent performs is visible in the LIVE run (the in-memory `trace` renders it via `BrainTimeline` with no allow/deny filter) but is **never persisted**, so on reopen/reload only assistant prose survives and every tool + memory step disappears. Two sub-gaps: (a) tool/recall/learn/reconcile steps live only in `brain-embedded/src/brainRunStore.ts`'s in-memory `trace` (the chat tables persist only assistant text); (b) the Evermind "learn" step shown mid-run is a client-side HEURISTIC guess (`brainRunStore.ts` ~864-880), not a truthful report of the server-side `learnFromBrainTurn` write (which runs in `waitUntil` and returns nothing).
> - **No migration required:** `brain_chat_messages.role` is `varchar(16)` (no CHECK) and `metadata` is nullable `text`, so steps can persist as `role:'tool'` rows with the step payload (bounded) in `metadata`.
> - **The vertical slice (4 coordinated edits, browser-verify each):** (1) `brainRunStore.ts runLoop` ŌĆö alongside each `pushTrace` for a tool/recall/learn/reconcile step, `persistence.sendMessages(chatId,[{ role:'tool', content:<compact>, metadata: JSON.stringify({kind:'step', category, label, args, boundedResult}) }])`; (2) `useBrainConversation.ts` ŌĆö BOTH seed builders (~344, ~379) must EXCLUDE `role:'tool'` messages so a persisted step never re-enters the model transcript (an orphaned `tool` message 400s strict vendors like Gemini) ŌĆö this is the correctness-critical edit; (3) `packages/brain-ui/src/timelineModel.ts buildSettledTimeline` (~106-197) ŌĆö reconstruct tool/recall/learn/reconcile nodes from persisted `role:'tool'` metadata (renderer `BrainTimeline.tsx` already handles those node kinds); (4) the chat-bubble message loader/renderer (`frontend/.../brain/BrainPanel.tsx`) must ROUTE `role:'tool'` messages to timeline nodes, not render them as empty chat bubbles. (5) Optional truthful-learn: have the send-messages response report whether `learnFromBrainTurn` landed and drive the `learn`/`reconcile` step off that instead of the local heuristic.
> - Unblocks: durable, reload-surviving visibility of what the Brain agent read/changed/remembered ŌĆö the operator's "we should be seeing that in the live chat" expectation. Deferred from the A+B model-floor/auto-compaction pass because the seed-exclusion + renderer-routing correctness needs browser verification and a half-done version risks broken `tool` bubbles or orphaned-tool 400s.

### ­¤öÉ Login "CORS error for everyone" ŌĆö investigate the Cloudflare-level trigger

> Reported 2026-07-09: users could not log in (any method); browser console showed `net::ERR_FAILED` + "No 'Access-Control-Allow-Origin' header is present" on **every** endpoint (`/health`, `/api/auth/legal/current`, `/api/auth/web/login`). From outside the worker is healthy ŌĆö curl (incl. fully browser-shaped requests) gets `Access-Control-Allow-Origin: *` on every method + preflight, so this is NOT a worker CORS-config bug. A headerless response reaching the browser means the response came from a **Cloudflare-level failure** (Worker exception 1101, CPU/mem 1102, or a WAF/Bot-Fight/rate-limit challenge page), which is served without the worker's CORS headers.
> - **Mitigated (api 2026.7.67):** the top-level `fetch` handler (`api/src/index.ts`) now wraps `buildApp(env).fetch(...)` in try/catch and returns a **CORS'd JSON 500** ŌĆö so a construction/dispatch throw is now a readable error in the browser instead of a phantom "CORS error for everyone". This converts the invisible failure into an observable one; it does not remove the trigger.
> - **Still open ŌĆö find the actual trigger:** (a) check Cloudflare dashboard ŌåÆ Workers logs / Logpush for `1101/1102` exceptions and the `[fetch:top-level]` / `[unhandled]` console lines around the incident window; (b) check `api_error_log` table + the Product Quality pillar for the real 500 message; (c) confirm no **Bot Fight Mode / WAF / rate-limit** rule is challenging browser (cookie-bearing, residential-IP) traffic while letting datacenter curl through; (d) rule out a transient **Neon** connection-construction failure in `buildDatabase`/`buildApp` under cold-start/connection pressure. Unblocks: root-cause + permanent fix for the login outage class.
