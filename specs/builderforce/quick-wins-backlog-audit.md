# Quick Wins Discovery & Classification — BuilderForce.AI Backlog
- Generated by PRD (#223) — Low-Hanging Fruit
- Scope: All tasks tagged **low or medium priority** and in **backlog/** **to-do/** **open** status (according to FR-1).
- Date: 2025-06-18; Project: BuilderForce.AI (id 11); Ticket: #223
- Sources: Multi-lane board scan (retro + backlog + all statuses). 155 backlog tasks (all medium/low) plus low-priority tasks surfaced across statuses. Detailed board state was fetched (first scan). Each eligible task was evaluated for heuristics:
  - **Effort**: ≤2 SP or explicitly smaller-tagged (S/XS) or unscoped stories/minor refactors (where we inferred ≤4h).
  - **Adequate clarity**: Checks for explicit acceptance criteria or a clearly stated output; e.g., `builtin_chats_consolidate(...)`/`builtin_brain_update(...)` tasks with bulleted steps and acceptance lines.
  - **No blockers**: Confirmed by task read (checks description and linked tickets; none had blocker tags or dependent open issues).
  - **Single-team**: Confirmed by checklist (assigned to a single team/event owner / free of multi-team gates). Cross-team heuristics were explicitly checked against the descriptions.
  - **Self-contained**: Verifies no waiting on design/legal/arch decision (verified against descriptions and linked tickets/mepgs; patterns like design doc checks were rejected).
- **Quick Win Score (QWS)** counts total heuristics satisfied: 1–5. Scoring thresholds:
  - Tier 1: 4–5 QWS — immediate pickup (to-do / picked up now)
  - Tier 2: 2–3 QWS — batch or plan for pickup (recommend batching with sprint or lanes)
  - Tier 3: ≤1 QWS — not included (insufficient heuristics met)
- The output groups items by **theme** (PE/bug docs/refactors/process; exp migrates; trust/features; health/reporting; governance/ops; external SaaS reads/writes; developer experience) to enable batch work (FR-5). Each row includes: Task ID (from board), Title, Current priority, QWS (out of 5), Tier, Theme, Rationale (one line), Recommended Action (pick up now/batch with sprint/needs clarification). Ambiguous tasks (where scope is not explicit and may cross themes or interact with other sub-features) appear in a separate “Needs clarification” section (FR-6).
